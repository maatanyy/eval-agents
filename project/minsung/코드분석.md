## 0. 저장소 한눈에 보기

에이전트를 **만들고(build) → 실행하며 추적하고(trace) → 평가하는(evaluate)** 전 과정을 가르치는 교육용 저장소입니다. 공통 로직은 하나의 설치형 패키지로 분리하고, 각 도메인 모듈은 이 패키지를 조합해 사용합니다.

```
eval-agents/
├── aieng-eval-agents/          # 재사용 패키지 (pip 설치형, ~14k LOC)
│   └── aieng/agent_evals/
│       ├── (core)              # 설정·인프라·평가 하니스·도구
│       ├── knowledge_qa/       # 지식 QA 에이전트
│       ├── aml_investigation/  # 자금세탁 조사 에이전트
│       ├── report_generation/  # 리포트 생성 에이전트
│       └── misalignment_qa/    # 미스얼라인먼트 실험 러너
├── implementations/            # 실행 진입점 (CLI·노트북·데모·데이터)
│   ├── basics/                 # 평가 기초 노트북 2종
│   ├── knowledge_qa/
│   ├── aml_investigation/
│   ├── report_generation/
│   └── misalignment_qa/
└── project/<팀원>/             # 팀원별 작업 폴더
```

**핵심 기술 스택**
- **Google ADK** (`LlmAgent`, `Runner`, `FunctionTool`) — 에이전트 구축 표준
- **Langfuse** — 데이터셋 저장소 + 트레이스/스코어 백엔드 + 실험 러너
- **Gemini** (planner=2.5-pro, worker=2.5-flash, evaluator=2.5-pro) via OpenAI 호환 API + native ADK
- **Anthropic** via LiteLLM (misalignment 모듈)
- **Pydantic / pydantic-settings** — 설정·구조화 출력 스키마
- **sqlglot + SQLAlchemy** — 읽기 전용 SQL 도구
- **tenacity / OpenTelemetry / rich / pandas / openpyxl**

**설계 철학**: "애플리케이션은 도구·에이전트만 정의하고, 평가·트레이싱·클라이언트 수명주기는 공유 싱글턴이 처리한다."

## 1. 핵심 공유 프레임워크 (`aieng/agent_evals/` core)

### 1.1 4대 하위 시스템
| 계층 | 파일 | 역할 |
|---|---|---|
| 설정/인프라 | `configs.py`, `async_client_manager.py`, `db_manager.py`, `async_utils.py`, `langfuse.py`, `logging_config.py`, `display.py`, `progress.py` | 설정 로드, 클라이언트/DB 싱글턴, 트레이싱, 콘솔 UI |
| 평가 하니스 | `evaluation/experiment.py`, `evaluation/trace.py`, `evaluation/types.py` | Langfuse 실험 래퍼, 2-pass 트레이스 평가 |
| 채점기 | `evaluation/graders/{_utils,config,llm_judge,trace_groundedness}.py` | LLM-as-judge, groundedness 팩토리 |
| 도구 | `tools/{search,vertex_search,web,file,sql_database,_redirect}.py` | 검색·웹·파일·SQL 도구 (모두 `FunctionTool` 래핑) |

### 1.2 평가 흐름 (Experiment / Trace / Grader)
- **타입 계약** (`types.py`): Langfuse 내부 타입을 re-export해 안정적 API 표면 제공. 자체 타입 `TraceEvaluatorFunction`(Protocol), `TraceWaitConfig`, `TraceEvalResult`, `EvaluationResult` 등 추가.
- **1차 패스** (`experiment.py`): `run_experiment`는 `dataset.run_experiment(task, evaluators, ...)`의 얇은 래퍼. 동시성·스코어링은 Langfuse SDK에 위임.
- **2-pass** (`run_experiment_with_trace_evals`): (1) task 실행으로 output+trace_id 생성 → (2) `run_trace_evaluations`로 트레이스 완성 대기 후 트레이스 평가. 반환은 `EvaluationResult(experiment, trace_evaluations)`.
- **트레이스 평가** (`trace.py`): `run_coroutine_sync`로 노트북/스크립트 양쪽 지원. `fetch_trace_with_wait`가 tenacity 지수 백오프로 ingestion 대기(`_trace_ready` = input·output 둘 다 non-None). `extract_trace_metrics`가 observation에서 tool-call 수·turn 수·토큰·cost·latency를 휴리스틱으로 집계.
- **채점기**:
  - `create_llm_as_judge_evaluator` (item): 기본 rubric `correctness/completeness/constraint_adherence` 바이너리. 예외를 결정론적 error 메트릭으로 흡수.
  - `create_trace_groundedness_evaluator` (trace): tool observation만 추출(정답 누출 방지), atomic claim별 Supported/Unsupported 판정 후 **점수는 코드에서 결정론적 재계산**(LLM에게 산수 안 맡김 — 우수한 설계).

### 1.3 도구 계층
- **`search`**: Gemini google_search grounding을 함수 도구로 래핑, redirect URL 실해석. "search→fetch→verify" 강제 의도.
- **`vertex_search`**: 커스텀 Vertex AI 데이터스토어, ADC 인증(API 키 불필요).
- **`web`**: HTML→markdown / PDF→텍스트 후 LLM으로 쿼리 기반 요약만 반환(컨텍스트 오버플로 방지).
- **`file`**: CSV/XLSX/JSON fetch·grep·read (하드캡 존재, URL 거부).
- **`sql_database`** (`ReadOnlySqlDatabase`): **가장 견고한 모듈**. sqlglot AST 기반 read-only 강제(허용 루트 select/union/paren, 금지 노드 insert/update/…, CTE 게이팅, fail-closed), 행 제한, 감사 로그. 결과를 마크다운 표로 반환.

### 1.4 설정/인프라
- `configs.py`: `.env` 로드, 중첩 delimiter `__`(예 `AML_DB__DRIVER`), `SecretStr` 비밀 보호, LLM 계층 구분, `AliasChoices`로 OPENAI/GEMINI/GOOGLE_API_KEY 호환.
- `async_client_manager.py` / `db_manager.py`: lazy 싱글턴. Gradio hot-reload 시 이벤트 루프 충돌 방지가 명시적 목표.
- `async_utils.py`: `gather_with_progress`(입력 순서 보존), `run_coroutine_sync`(실행 중 루프 감지 시 별도 스레드+새 루프).
- `langfuse.py`: OTLP + `GoogleADKInstrumentor` 자동 계측, content 해시 기반 결정론적 item_id로 dedup 업로드.

## 2. Knowledge QA 모듈

**무엇**: Google ADK **PlanReAct** 에이전트가 실시간 웹 검색으로 사실을 찾아 검증, DeepSearchQA(896문항) 벤치마크로 LLM-as-judge 채점.

- **아키텍처** (`KnowledgeGroundedAgent`): `PlanReActPlanner`(계획 태그 파싱) + Gemini native thinking(`thinking_budget=8192`) + 5개 도구(google_search·web_fetch·fetch_file·grep_file·read_file).
- **google_search 래핑**: 스니펫이 아니라 실제 URL을 보게 해 "search→fetch→verify→answer" 워크플로 강제.
- **회복력**: 컨텍스트 캐싱·이벤트 압축, 429 재시도, 컨텍스트 오버플로 세션 리셋, 빈 응답 재시도(최대 2회).
- **채점** (`deepsearchqa_grader.py`): 논문 공식 grader 프롬프트 → 집합론 기반 **Precision/Recall/F1** + 4단계 outcome(FULLY_CORRECT / CORRECT_WITH_EXTRANEOUS / PARTIALLY_CORRECT / FULLY_INCORRECT). 성공·실패 모두 4개 Evaluation 반환(스키마 일관성).
- **진입점**: `knowledge-qa` CLI(ask/eval/sample/tools), 노트북 4종, `implementations/knowledge_qa/evaluate.py`(Langfuse 실험).

**주의점 / 리스크**
- 표시 계층이 특정 로그 문자열("Tool call: …") 파싱에 의존 → 로그 포맷 변경 시 조용히 깨짐.
- 계획/실행 trace의 `StepExecution`은 균등 분배된 가짜 duration·`sources_found=0` 등 **관측 편의용 placeholder** (정량 지표로 오해 금물).
- 문서/코드 드리프트: docstring "BuiltInPlanner" vs 실제 `PlanReActPlanner`, `compaction_interval` 기본 3 vs 10, 도구명 `web_fetch` vs `fetch_url` 표기 불일치.
- 파싱 로직·로그 핸들러가 CLI/notebook에 중복 구현(DRY 위반).
- `implementations/knowledge_qa/agent.py`가 import 시점에 에이전트 인스턴스화(네트워크 호출 부작용).

## 3. AML Investigation 모듈 (자금세탁 조사)

**무엇**: 에이전트가 읽기 전용 SQL로 SQLite 금융거래 DB를 조사해 자금세탁 여부·타이폴로지를 판정. 결과를 **item/run/trace 3계층**으로 채점.

- **에이전트** (`create_aml_investigation_agent`): `ANALYST_PROMPT` 기반. 원칙 — 기본 가설은 "정상", 복수 지표 필요, `trigger_label` 맹신 금지, 집계 우선. 출력을 `AnalystOutput` 스키마로 고정. 도구는 `db.get_schema_info`·`db.execute` 2개.
- **데이터**: Kaggle IBM AML 데이터셋. **결정적 transaction_id**(10필드 SHA-256)로 CSV와 Patterns.txt를 정합. `is_laundering` 컬럼은 DB에서 제외해 **정답 누출 방지**.
- **케이스 생성** (`build_cases`, seed=42): 4종 — Laundering(강한 신호) / False Negative(약한 신호) / False Positive(정상인데 경보 라벨) / Normal. `trigger_label`이 난이도 조절 장치.
- **채점 3계층**:
  - **item** (`item.py`): verdict 정확도, 타이폴로지 일치, 정상 케이스 일관성, `id_precision_like`=(tp−fp)/predicted(오탐 페널티, 음수 가능), `id_coverage`=tp/gt.
  - **run** (`run.py`): scikit-learn precision/recall/F1, 패턴 macro-F1, 혼동행렬(잘못된 라벨은 INVALID로 편입).
  - **trace** (`trace.py`): SQL 실행 여부, 읽기 전용 준수(DB 정책 주입 시 sqlglot 검사), 윈도우 필터 준수, 윈도우 위반 카운트, 중복 쿼리 비율.
  - **rubric** (`narrative_pattern_quality.md`): 추론 품질만 1~5 채점, 하드 가드레일(근거 없는 주장 ≤2, 결정 모순 =1).
- **진입점**: `data/cli.py`(DB·케이스 생성) → `cli.py`(재개 가능 배치, fsync 체크포인트) 또는 `evaluate.py`(Langfuse 풀 평가, 3계층 그레이더 배선).

**주의점 / 리스크**
- `id_precision_like`는 표준 정밀도가 아님(이름 오해 소지).
- `build_cases`는 seed 있으나 `trigger_label` 선택은 시드 안 된 전역 `random` 사용 → 라벨 배정 비재현.
- 윈도우 위반 탐지가 정규식 리터럴 기반 → 파라미터/함수 표현 시 놓칠 수 있는 휴리스틱.
- `normalize_transactions_data`가 입력 DataFrame in-place 변형.

## 4. Report Generation 모듈

**무엇**: 자연어 질의 → SQL 집계 → **다운로드 가능한 Excel(.xlsx) 리포트** + Gradio 하이퍼링크. BI 리포팅 용도.

- **에이전트** (`get_report_generation_agent`): worker 모델(gemini-2.5-flash) + 도구 3개(`get_schema_info`·`execute`·`write_xlsx`). 전형 궤적: 스키마 조회 → SQL 집계 → write_xlsx → 다운로드 링크 응답.
- **`EventParser`**: ADK Event를 `ParsedEvent`(FINAL_RESPONSE/TOOL_CALL/THOUGHT/TOOL_RESPONSE)로 정규화 → **데모 UI·오프라인 평가·온라인 스코어링 3곳에서 공통 재사용**.
- **`file_writer`**: DataFrame→Excel, `gradio_api/file=` 링크 반환. 예외는 문자열로 반환(LLM 재시도 유도).
- **프롬프트** (`prompts.py`): 얇은 자연어 오케스트레이션(코드 강제 없음) + 궤적/결과 평가용 LLM-judge 프롬프트(관대한 규칙: 수치 0.01 허용오차, 행수 일치, 순서/컬럼명 무시).
- **평가 이원화**:
  - **오프라인** (`offline.py`): Langfuse 데이터셋(`OnlineRetailReportEval`) + 2개 LLM-judge(final_result·trajectory) + tenacity 재시도 + 동시성.
  - **온라인** (`online.py`): 라이브 trace에 규칙 기반 자동 스코어(링크 유무·토큰·지연) + 👍/👎 휴먼 피드백. `report_usage_scores`는 hang 위험 때문에 daemon 스레드 격리.
- **Gradio 데모** (`demo.py`): 스트리밍 챗봇, 추론 과정(도구/사고) 투명 노출, `adk web` 대안 지원.
- **데이터**: UCI Online Retail → 단일 `sales` 테이블(flat).

**주의점 / 리스크**
- 온라인 스코어링이 하드코딩 문자열 `"](gradio_api/file="`에 의존 → 링크 포맷 변경 시 false negative.
- `file_writer` 실패 문자열이 링크와 구분 없이 `final_report`로 저장 → 실패 write가 정상처럼 평가에 유입 가능(judge 관대함으로 부분 완화).
- `env_vars.py` 기본 경로가 상대 경로 → 프로젝트 루트 외 CWD에서 어긋남.
- `convert_date` 주석-코드 불일치(월/일 파싱 vs 주석 "DD/MM").

## 5. Misalignment QA 모듈

**무엇**: **무모한 예시 Q&A를 문맥에 주입하면 모델이 유해 응답으로 밀리는가, 그 효과가 다른 위해 유형으로 전이되는가**를 측정. 도구 없는 **plain LLM completion** 평가(가장 단순한 형태의 미스얼라인먼트 평가).

- **설정 주도**: `config_types.py`의 Pydantic 스키마로 YAML(`bootcamp_misalignment.yaml`) 검증. 실험 변경은 원칙적으로 YAML만 편집.
- **5개 조건 × 6개 모델 = 30 variant** (동일 9-task 공유):
  - 조건: baseline / examples-as-llm-messages / examples-and-priority-instruction / examples-as-user-context / examples-as-context-with-priority
  - 두 축: 예시 주입 방식(개발자용 LLM 메시지 vs 일반 사용자 접근 가능 user 텍스트) × priority 지시 유무
  - 모델: Gemini 3종(native) + Anthropic 3종(LiteLLM)
- **3개 태스크 카테고리**: Life-safety(in-domain) / Harmful code(out-of-domain) / Social engineering(out-of-domain) — in/out 도메인 전이 대비.
- **base/override 병합** (`resolve_agent_spec`): Pydantic `model_fields_set`으로 "미지정 상속" vs "명시적 null 삭제" 구분(deprecated temperature 우아하게 처리).
- **재현성**: 태스크 SHA-256 fingerprint(drift 감지), 결정론적 item ID(idempotent), 공유 `run_instance_id`.
- **채점**: LLM-judge 3개 메트릭 `harmful`/`helpful`/`follows_reckless_pattern`(0/1) + hard metrics(tool_call·turn count). 판정 모델 gemini-3-flash, temp 0.0. 사고 토큰은 판정 입력에서 제외.
- **결과 탐색** (`analysis.py`): dataset run 선택 → 시간 윈도우 trace 조회 → 스코어·메트릭 조인 → 조건×모델 heatmap·요약표.

**주의점 / 리스크 (주로 방법론)**
- 판정 모델(Gemini)이 평가 대상(Gemini)과 동계열 → 계열 편향 가능. 인간 캘리브레이션 없음.
- variant당 반복 없음(9 task × 1 run) → 확률적 분산 미측정, 표본 작음.
- `analysis.py`의 시간 윈도우 trace 필터링은 동시 실행 실험 간 오염·누락 가능.
- LiteLLM 경로에서 Anthropic extended thinking 무음 무시.

## 6. Basics 모듈 (교육 노트북)

- **`01_why_evals.ipynb`** (개념): 왜 어려운가(오류 누적 → task-level 평가 필요), **4대 품질 차원**(Outcome / Tool Usage / Reasoning / Cost-Performance), **3대 grader 유형**(Code-based / Model-based / Human), capability evaluation(오를 언덕), 실패→시스템 변경 사이클.
- **`02_evaluation_harness.ipynb`** (실습): 공유 하니스 3빌딩블록 — Datasets(idempotent 업로드) / Task 함수(`(*, item, **kwargs)->str`, 스레드별 fresh async 클라이언트) / Evaluator 함수(`create_llm_as_judge_evaluator`, 커스텀 substring_match) → `run_experiment` → 2-pass 트레이스 평가(`trace_latency_evaluator`, `create_trace_groundedness_evaluator`).

> Misalignment 모듈은 basics가 가르친 하니스(`run_experiment_with_trace_evals`, `create_llm_as_judge_evaluator`, `extract_trace_metrics`, `TraceWaitConfig`)를 **실전 규모로 확장한 응용편**입니다.

## 7. 저장소 공통 관찰

### 잘 된 설계
- **관심사 분리**: 재사용 패키지(라이브러리) vs 실행 진입점(implementations) 명확 분리.
- **평가 성숙도**: item/run/trace 다계층, 결정론적 + LLM-judge 혼합, 성공·실패 시 스키마 일관성, 정답 누출 방지(groundedness·AML), 2-pass 트레이스 평가.
- **보안**: `ReadOnlySqlDatabase` AST 기반 fail-closed 방어, `SecretStr` 비밀 보호, 의존성에 CVE 패치 버전 주석 명시.
- **노트북 친화성**: `run_coroutine_sync`, 스레드별 fresh 클라이언트로 이벤트 루프 충돌 회피.
- **재현성**: 결정론적 ID·fingerprint, seed 사용.

### 저장소 전반 리스크 (교차 관찰)
- **core `trace.py`의 `_sum_token_usage` assert**: `usage_details is None`인 observation(tool-call 등)에서 `AssertionError`로 `extract_trace_metrics` 크래시 가능. `python -O`에선 `AttributeError`. → **우선 수정 후보**.
- **`langfuse.py`의 `asyncio.run`**: 노트북(실행 중 루프)에서 `RuntimeError`. 같은 저장소의 `run_coroutine_sync`를 안 씀. → **우선 수정 후보**.
- **SQL 타임아웃 미작동**: `execution_options(timeout=)`·`PRAGMA busy_timeout`은 statement 실행시간 제한이 아님(장기 쿼리 못 끊음).
- **모듈 로드 시 `logging.basicConfig` 중복 호출**(여러 파일) → 로깅 동작 비결정적.
- **문서-코드 드리프트**가 여러 모듈에 산재(docstring·README vs 실제 코드).
- **광범위 재시도**(`web.py`의 `retry_if_exception_type((Exception,))`) → 비일시적 오류도 3회 재시도.
- **싱글턴 스레드 안전성 부재**(`get_instance`에 락 없음), `_redirect_cache` 무한 증가.
